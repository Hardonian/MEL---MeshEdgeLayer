# Hosted vs self-hosted strategy memo (internal)

## Principle

MEL’s **credible wedge** is local-first **operator truth**: evidence, degraded semantics, and governable control paths. Anything “hosted” must not become a hidden trust migration that contradicts that story.

## Self-hosted (default credible path)

- Full data custody on operator hardware.
- No mandatory cloud control plane for base viability (see [AGENTS.md](../../../AGENTS.md)).
- Fits clubs, labs, integrators, and security-minded teams.

## Hosted command center (staged / speculative)

If offered in the future, it must:

- preserve explicit **data boundary** language (who stores what, for how long);
- avoid implying stronger RF or routing proof than the self-hosted binary;
- not silently subsume canonical truth into a vendor dashboard narrative.

## Open-core alignment

- **Open:** ingest truth, incident surfaces, local audit posture that already ship.
- **Commercial leverage:** support, packaging, governance depth, onboarding for teams — only as implemented; see [EDITION_PACKAGING.md](../../product/EDITION_PACKAGING.md).

## Risks to avoid

- “Cloud makes it enterprise” without tenancy, security, and ops evidence.
- Telemetry or central aggregation that undermines privacy-first positioning without explicit opt-in documentation.
