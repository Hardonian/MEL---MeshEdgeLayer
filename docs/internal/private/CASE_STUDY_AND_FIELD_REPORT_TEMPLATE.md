# Case study and field report template (internal)

Use for **anonymized** or approved public narratives. Do not claim RF success, routing proof, or “AI diagnosis” without persisted evidence citations.

## Field report (operator-internal)

**Date / site type / role**

**Environment**

- Hardware / OS / MEL version (commit or tag)
- Transports in use (serial / TCP / MQTT) and topology description (high level)

**Objective**

- What you tried to prove or operate

**Observed (evidence-backed)**

- Timestamps, doctor output, metrics endpoints, screenshots tied to DB state

**Inferred / estimated (label clearly)**

- Hypotheses not yet backed by persisted records

**Unknown**

- Explicit gaps

**Actions and lifecycle**

- Control actions: submission vs approval vs execution vs audit (as applicable)

**Outcome**

- What improved, what did not, what would change in config or process

**Artifacts**

- Proofpack / export / support bundle references (redacted IDs)

---

## External case study (public-safe)

**Title** — one line, no superlatives

**Context** — organization type without sensitive detail

**Problem** — incident ambiguity, trust, or operational memory (specific)

**Approach** — how MEL was deployed (self-hosted, transports used)

**Results (bounded)** — metrics you can prove (e.g. time to bundle export, fewer ambiguous states) — avoid unverifiable “downtime reduced X%”

**What MEL did not do** — routing, coverage proof, unsupported transports

**Quote** — optional, attributable
